# React Architect — Architecture Constitution

> **This document is the source of truth for the React Architect experience.**
>
> Every implementation decision must support this vision.
> If code conflicts with this document, the code is wrong.

---

# What is React Architect?

React Architect is **not a website**.

It is an interactive software experience.

Users should feel like they are entering a carefully engineered digital workspace—not navigating between web pages.

The opening sequence is not a hero section.

It is an arrival.

The workspace is not loaded.

It is revealed.

---

# Core Philosophy

Everything in React Architect is built around one idea:

> **Architecture is created through relationships.**

The experience should communicate:

* Construction
* Structure
* Composition
* Transformation
* Organization
* Precision
* Systems

Never decoration.

Never visual noise.

---

# Product Personality

React Architect should feel:

* Intelligent
* Precise
* Calm
* Premium
* Confident
* Engineered

It should never feel:

* Playful
* Toy-like
* Overly futuristic
* Cyberpunk
* Game-like
* Flashy

---

# The Modular Principle

Everything is built from reusable modules.

Not objects.

Not scenes.

Modules.

Examples include:

* Structural rails
* Precision joints
* Thin architectural frames
* Glass plates
* Connector nodes

Every visual state uses the exact same modules.

Nothing new appears.

Nothing disappears.

Everything rearranges.

---

# The Living Artifact

There is one central artifact.

It is not a logo.

It is not branding.

It is a living architectural object.

This artifact represents the heart of React Architect.

Every major animation is a transformation of this artifact.

The artifact should always feel:

* balanced
* intelligent
* engineered
* responsive

Never random.

Never decorative.

---

---

# Typography Philosophy

Never use text geometry.

Never import logo meshes.

The title "React Architect" is physically constructed from the same modular pieces used throughout the experience.

The letters are temporary arrangements of the modules.

When the typography transforms, nothing is replaced.

The same pieces simply find a new purpose.

---

# Motion Philosophy

Motion exists to explain architecture.

Every movement must communicate intent.

Good motion:

* Assembly
* Alignment
* Compression
* Expansion
* Connection
* Construction
* Reorganization
* Balance

Bad motion:

* Random spinning
* Floating for no reason
* Particle explosions
* Visual effects without meaning
* Continuous unnecessary movement

---

# Interaction Philosophy

The cursor should influence the object.

Not control it.

Hover should feel like curiosity.

The artifact acknowledges attention.

Possible responses:

* subtle structural shifts
* internal movement
* light redistribution
* magnetic attraction
* layers separating slightly

The object never behaves like a toy.

---

# Camera Philosophy

The camera is part of the storytelling.

Never hard-cut.

Never teleport.

Every movement must feel intentional.

The user should feel like they are physically travelling through architecture.

---

# Portal Philosophy

The portal is not a new object.

The portal is the artifact itself.

The artifact unfolds.

Structural members extend.

Frames separate.

Space opens.

The center becomes an impossible gateway.

The camera travels through that gateway.

---

# Workspace Philosophy

The workspace is hidden inside the artifact.

The portal is the transition between compressed architecture and usable architecture.

The user should feel:

"I didn't switch screens."

"I entered the object."

---

# Visual Language

Materials:

* anodized aluminium
* brushed metal
* architectural glass
* matte graphite
* subtle emissive lighting

Avoid:

* chrome overload
* rainbow gradients
* excessive bloom
* noisy textures
* sci-fi clichés

Lighting should feel like a premium product showroom.

---

# Design Rules

1. Build systems, not scenes.
2. Everything is modular.
3. Every state is built from the same modules.
4. Never replace geometry.
5. Never fake transitions.
6. Every transformation is physical.
7. Motion explains architecture.
8. Simplicity beats complexity.
9. Premium beats flashy.
10. Empty space is as important as geometry.

---

# Technical Principles

Preferred stack:

* React
* React Three Fiber
* Drei
* GSAP
* Instanced Meshes where appropriate
* Data-driven layouts

Avoid hardcoded animations.

Prefer:

Modules → Target Layouts → GSAP interpolation.

Each module should know where it belongs in multiple layouts rather than creating separate objects.

---

# AI Development Rules

When contributing code:

* Do not over-engineer.
* Do not introduce unnecessary abstraction.
* Do not use placeholder hacks.
* Explain architectural decisions before implementation.
* Keep components composable.
* Prefer reusable systems over one-off effects.
* Every new feature must support future transformations.

If a feature breaks the illusion that everything is built from the same modular system, redesign it.

---

# The North Star

A first-time user should experience the intro and think:

> "I've never seen a React product introduced like this."

The experience should feel less like opening a website and more like watching architecture come alive.
